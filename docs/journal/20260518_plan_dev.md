# Mosaic.jl — Live Coding Musical en Julia

> **Note** : `Mosaic` est un nom de travail. À renommer si tu trouves mieux
> (suggestions : `Caudal.jl`, `Cyclos.jl`, `Polyrythme.jl`).

---

## 1. Vision et contexte

### Ce qu'on construit

Un environnement de **live coding musical en Julia**, avec :

1. Un **DSL de patterns** inspiré de TidalCycles (algèbre de patterns comme
   fonctions du temps vers events), mais idiomatique Julia (multiple dispatch,
   string macros, types paramétrés).
2. Un **scheduler temps réel** qui query les patterns en boucle et envoie des
   events OSC à un backend de synthèse (scsynth/SuperDirt en Phase 1).
3. Une **TUI** pour éditer, évaluer en live, et voir le statut (cycle courant,
   patterns actifs, latence).
4. (Phase 3) Une **intégration reservoir computing** où un ESN peut être branché
   comme source de pattern de première classe, combinable avec les patterns
   manuels via les mêmes opérateurs.

### Pourquoi

L'écosystème live coding actuel (TidalCycles en Haskell, Strudel en JS,
SuperCollider en sclang) ne couvre pas Julia. Or Julia a des atouts spécifiques
pour ce use case :

- **Multiple dispatch** rend la composition d'opérateurs sur patterns triviale
  (vs typeclasses Haskell).
- **JIT + Revise.jl** donnent un vrai hot-reload avec persistance d'état.
- **Écosystème SciML / ReservoirComputing.jl** ouvre une voie unique : générer
  de la musique avec des systèmes dynamiques entraînables, en tant que citoyens
  de première classe du DSL.

### Hors-scope (explicite)

- Pas d'implémentation propre du DSP audio en Phase 1-3. On délègue à scsynth.
- Pas de GUI graphique. TUI uniquement.
- Pas d'interface web.
- Pas d'intégration DAW (VST/AU plugins).
- Pas de support MIDI in/out en Phase 1 (Phase 4 éventuellement).

## 5. Modules détaillés

### 5.1 `core.jl` — types fondamentaux

**Responsabilité** : définir les briques de base.

**Types publics** :

```julia
"""
    Event{T}

Un événement temporel : une valeur de type `T` placée dans un arc temporel
[start, stop). Les temps sont en `Rational{Int64}` (cycles), exacts.
"""
struct Event{T}
    start::Rational{Int64}
    stop::Rational{Int64}
    value::T
end

"""
    Pattern{T}

Un pattern est une fonction qui, étant donné une fenêtre temporelle [s, e),
renvoie tous les events tombant dedans, triés par start.
Sémantique : `Pattern{T} ≈ (Rational, Rational) -> Vector{Event{T}}`.
"""
struct Pattern{T}
    query::Function  # (Rational, Rational) -> Vector{Event{T}}
end

# Sucre : un Pattern est callable
(p::Pattern)(s::Rational, e::Rational) = p.query(s, e)
```

**Fonctions publiques** :
- `query(p::Pattern, s, e) -> Vector{Event}` (alias `p(s, e)`).

**Invariants à respecter** :
- Les events renvoyés par `query` sont **clip** à la fenêtre demandée
  (un event qui chevauche est tronqué, pas exclu).
- Pas de mutation : un `Pattern` est immutable (la `Function` interne capture
  son état par closure si besoin).
- L'ordre temporel des events renvoyés est croissant sur `start`.

**Acceptance criteria** :
- [ ] Les types compilent et sont exportés depuis `Mosaic`.
- [ ] Tests : construction d'un `Event`, query d'un `Pattern` trivial.

### 5.2 `combinators.jl` — combinateurs de base

**Responsabilité** : fournir les combinateurs canoniques inspirés de Tidal.

**Fonctions publiques** :

| Combinator | Signature | Sémantique |
|---|---|---|
| `pure` | `pure(v::T) -> Pattern{T}` | La valeur `v` à chaque cycle entier. |
| `silence` | `silence(::Type{T}) -> Pattern{T}` | Pattern vide. |
| `fast` | `fast(n::Real, p::Pattern{T}) -> Pattern{T}` | Compresse le temps de facteur `n`. |
| `slow` | `slow(n::Real, p::Pattern{T}) -> Pattern{T}` | Dilate (= `fast(1/n, p)`). |
| `rev` | `rev(p::Pattern{T}) -> Pattern{T}` | Inverse l'ordre des events dans chaque cycle. |
| `every` | `every(n::Int, f::Function, p::Pattern{T}) -> Pattern{T}` | Applique `f` au pattern tous les `n` cycles. |
| `stack` | `stack(ps::Vararg{Pattern{T}}) -> Pattern{T}` | Joue plusieurs patterns en parallèle. |
| `cat` | `cat(ps::Vector{Pattern{T}}) -> Pattern{T}` | Concatène : un pattern par cycle, en rotation. |
| `density` | alias de `fast` | Compatibilité Tidal. |

**Acceptance criteria** :
- [ ] Chaque combinator a un docstring avec un exemple.
- [ ] Tests : pour chaque combinator, au moins un test vérifie la sortie
  attendue de `query(0//1, 2//1)` (deux cycles).
- [ ] Property-based test : `fast(2, slow(2, p)) ≡ p` (à un near-equal près).

### 5.3 `algebra.jl` — opérateurs

**Responsabilité** : surcharger les opérateurs Base pour les Patterns.

**Méthodes à implémenter** :

```julia
import Base: +, -, *, /, ∘

# Pattern + Pattern : combine en intersectant les arcs, additionne les values
+(p::Pattern{T}, q::Pattern{T}) where {T<:Number}

# Pattern + scalar : décale toutes les values
+(p::Pattern{T}, x::T) where {T<:Number}
+(x::T, p::Pattern{T}) where {T<:Number}

# Idem pour -, *, /

# Composition fonctionnelle de transformations
∘(f::Function, g::Function)  # déjà dans Base, on étend pour Pattern -> Pattern
```

**Pour les patterns non-numériques** (ex: `Pattern{Symbol}` pour des noms de
samples), définir un combinator dédié `mask(p::Pattern{Symbol}, q::Pattern{Bool})`
qui ne joue les events de `p` que là où `q` vaut `true`.

**Acceptance criteria** :
- [ ] `p"0 2 4" + 12` transpose le pattern d'une octave (MIDI).
- [ ] Test des lois : commutativité de `+`, associativité.

### 5.4 `mininotation.jl` — parser

**Responsabilité** : convertir une string mini-notation en `Pattern`.

**Syntaxe à supporter (Phase 1)** :

| Syntaxe | Sémantique |
|---|---|
| `"bd hh sn hh"` | Séquence de 4 events dans un cycle. |
| `"bd ~ sn ~"` | `~` est un silence. |
| `"bd [hh hh] sn"` | Crochets = subdivision (les deux `hh` partagent le slot). |
| `"<bd sn cp>"` | Alterne entre éléments cycle après cycle. |
| `"bd*4"` | Répète l'event 4 fois dans son slot. |
| `"bd(3,8)"` | Rythme euclidien : 3 hits sur 8 steps. |
| `"bd!2"` | Réplique : étire l'event sur 2 slots. |

**API publique** :

```julia
"""
    parse_minino(s::String) -> Pattern{Symbol}

Parse une string mini-notation et renvoie un `Pattern{Symbol}`.
"""
function parse_minino(s::String) end

"""
    @p_str

String macro : `p"bd hh sn hh"` ≡ `parse_minino("bd hh sn hh")`.
"""
macro p_str(s) end
```

**Implémentation suggérée** : grammaire récursive simple (pas besoin de PEG en
Phase 1). Tokenize d'abord, puis construit l'AST récursivement pour gérer les
crochets et chevrons. Les rythmes euclidiens utilisent l'algorithme de Bjorklund.

**Acceptance criteria** :
- [ ] Tous les éléments de la table ci-dessus parsent correctement et
  produisent les events attendus sur `query(0//1, 1//1)`.
- [ ] Erreurs de parsing renvoient un message clair (avec position du token
  fautif).

### 5.5 `osc.jl` — encoding et transport OSC

**Responsabilité** : encoder des messages/bundles OSC, les envoyer en UDP.

**Spec OSC à implémenter** : OSC 1.0 (cf. https://opensoundcontrol.stanford.edu/spec-1_0.html).

**Types publics** :

```julia
"""
    OSCMessage(address::String, args::Vector{Any})

Un message OSC avec son address pattern et ses arguments typés.
Types supportés : Int32, Float32, String, Blob.
"""
struct OSCMessage end

"""
    OSCBundle(time::Float64, messages::Vector{OSCMessage})

Un bundle OSC avec time tag (NTP timestamp absolu en secondes).
"""
struct OSCBundle end

"""
    OSCClient(host::IPv4, port::UInt16)

Client UDP pour envoyer des messages/bundles OSC.
"""
struct OSCClient end
```

**Fonctions publiques** :
- `encode(msg::OSCMessage) -> Vector{UInt8}`
- `encode(bundle::OSCBundle) -> Vector{UInt8}`
- `send(client::OSCClient, msg_or_bundle)`

**Détails d'encoding (rappel)** :
- Address pattern : OSC-string (ASCII + null padding à un multiple de 4 bytes).
- Type tag string : `,` suivi des codes (`i` int32, `f` float32, `s` string,
  `b` blob), padded.
- Time tag : 8 bytes (32 bits seconds since 1900-01-01, 32 bits fractional).
- Bundle : `#bundle\0` + time tag + (size + message) pour chaque sous-message.

**Cible scsynth/SuperDirt** : format des messages attendu :
`/dirt/play` avec args `(s_name, sample, s, sound_name, n, note_idx, ...)`.
À documenter précisément en parsant le code SuperDirt ou en se référant à
la doc Tidal.

**Acceptance criteria** :
- [ ] Round-trip test : encode → decode → équivalent.
- [ ] Test d'intégration manuel : envoyer un `/dirt/play` à SuperDirt local,
  vérifier qu'un son sort.
- [ ] Bundle avec time tag dans le futur déclenche l'event au bon moment
  (mesuré avec `@elapsed`).

### 5.6 `scheduler.jl` — boucle temps réel

**Responsabilité** : gérer l'horloge, querier les patterns en avance, envoyer
les events en OSC bundles time-tagged.

**Types publics** :

```julia
"""
    Scheduler

État du scheduler de live coding.

# Champs
- `patterns::Dict{Symbol, Pattern}` : slots actifs (d1, d2, ...)
- `cps::Float64` : cycles par seconde (tempo)
- `lookahead::Float64` : seconds d'avance pour scheduling (default 0.05)
- `osc::OSCClient` : client OSC vers le backend
- `running::Threads.Atomic{Bool}`
- `t_start::Float64` : timestamp UNIX de démarrage
"""
mutable struct Scheduler end
```

**Fonctions publiques** :

```julia
"""
    start!(s::Scheduler)

Démarre la boucle de scheduling sur un thread dédié.
"""
function start!(s::Scheduler) end

"""
    stop!(s::Scheduler)

Stoppe proprement la boucle.
"""
function stop!(s::Scheduler) end

"""
    set_pattern!(s::Scheduler, slot::Symbol, p::Pattern)

Remplace (ou installe) le pattern dans le slot donné. Thread-safe.
C'est cette fonction qui réalise le "hot swap" du live coding.
"""
function set_pattern!(s::Scheduler, slot::Symbol, p::Pattern) end

"""
    hush!(s::Scheduler)

Stoppe tous les patterns (équivalent du `hush` de Tidal).
"""
function hush!(s::Scheduler) end

"""
    set_cps!(s::Scheduler, cps::Float64)

Change le tempo. Doit éviter les sauts de phase audibles.
"""
function set_cps!(s::Scheduler, cps::Float64) end
```

**Détails de la boucle principale** (à respecter) :

```
loop:
  now = time() - t_start
  now_cycles = now * cps
  end_cycles = (now + lookahead) * cps

  pour chaque (slot, pattern) dans patterns:
    events = pattern(rationalize(now_cycles), rationalize(end_cycles))
    pour chaque event:
      fire_time_unix = t_start + event.start / cps
      bundle = OSCBundle(fire_time_unix, [event_to_osc(event)])
      send(osc, bundle)

  sleep(min(lookahead / 2, 0.01))  # éviter double-fire
```

**Attention** :
- Maintenir un set d'events déjà envoyés pour la fenêtre courante (sinon double
  fire à chaque tour de boucle). Stratégie possible : mémoriser le `end_cycles`
  précédent et ne querier que `(last_end_cycles, end_cycles)`.
- L'accès à `patterns` doit être protégé (ReentrantLock ou Channel).
- Le `set_pattern!` doit prendre effet **au début du prochain cycle** pour
  éviter les glitches, pas immédiatement (option configurable).

**Acceptance criteria** :
- [ ] Démarrage / arrêt propre, pas de fuite de threads.
- [ ] Un pattern `p"bd"` joue 1 kick par cycle, à `cps=0.5` = 1 kick toutes
  les 2 secondes, jitter < 5 ms (mesuré côté scsynth via `/dirt/play`
  timestamps).
- [ ] `hush!` stoppe le son sous 1 cycle.
- [ ] Aucun double-fire constaté sur 10 minutes de fonctionnement.

### 5.7 `tui.jl` — interface terminal

**Responsabilité** : éditeur + status + logs, basé sur `TerminalUserInterfaces.jl`.

**Layout** :

```
┌─ Mosaic.jl ──────────────────────── cps: 0.5 │ cycle: 42 │ slots: d1 d2 ─┐
│                                                                          │
│ d1 = p"bd hh sn hh"                                                      │

│                                                                          │
│ # _                                                                      │
│                                                                          │
├──────────────────────────────────────────────────────────────────────────┤
│ [INFO] 14:23:01 Pattern d1 updated                                       │
│ [INFO] 14:23:15 Pattern d2 updated                                       │
└──────────────────────────────────────────────────────────────────────────┘
```

**Comportement** :
- Éditeur multi-ligne en haut.

- `Ctrl+H` : `hush!`.
- Status bar : cps, cycle courant, slots actifs, latence moyenne dernière sec.
- Logs : niveaux configurables (INFO par défaut, DEBUG via flag).

**Acceptance criteria** :
- [ ] L'éditeur accepte la saisie multi-ligne avec navigation flèches.
- [ ] Évaluation d'un bloc produit du son sous 200 ms.
- [ ] Les erreurs d'évaluation s'affichent dans la zone logs, ne crashent
  pas la TUI.

### 5.8 `reservoir.jl` — intégration ESN (Phase 3)

**Responsabilité** : exposer un ESN comme `Pattern`.

**Type public** :

```julia
"""
    ReservoirPattern{T}

Un pattern dont les events sont générés en faisant tourner un Echo State
Network. À chaque step temporel, l'état du reservoir est mis à jour, puis
une fonction de readout convertit l'état en valeur `T`.

# Champs
- `esn` : objet ReservoirComputing.ESN (ou compatible)
- `state` : Vector{Float64}, état courant
- `readout::Function` : state -> T
- `rate::Float64` : nombre d'events par cycle
"""
mutable struct ReservoirPattern{T} end
```

Doit implémenter la même interface `query(s, e)` qu'un `Pattern` standard, donc
**combinable avec les opérateurs de l'algèbre**.

**Trois readouts standard à fournir comme exemples** :
1. `note_readout` : quantize l'état sur une gamme → MIDI note.
2. `param_readout` : map continu vers un range (utile pour cutoff, etc.).
3. `spike_readout` : seuil → event symbolique ou silence.

**Acceptance criteria** :
- [ ] Un `ReservoirPattern` non-entraîné produit une séquence reproductible
  étant donné une seed.
- [ ] Test : `mask(reservoir_pattern, p"1 ~ 1 1")` produit les notes du
  reservoir mais uniquement aux pas marqués.
- [ ] Démo : entraîner un ESN sur un fichier MIDI simple, générer 4 cycles
  audibles.

## 6. Conventions de code

Toutes les conventions ci-dessous sont **non négociables**.

### Style Julia
- **Typing** : annotations sur toutes les fonctions publiques exportées.
- **Logging** : `@info`, `@warn`, `@debug`, `@error` de `Logging.jl`. Jamais
  de `println` en code de production (sauf dans scripts/demos).
- **Docstrings** : format standard Julia (triple quotes, `# Arguments`,
  `# Returns`, `# Examples`).
- **Config** : `struct` ou `NamedTuple` pour les hyperparamètres. Charger
  depuis `configs/default.toml`. Jamais de magic numbers dans le code.
- **Reproductibilité** : `Random.seed!()` documenté dans les scripts qui
  dépendent du hasard (reservoirs notamment).
- **Formatage** : `JuliaFormatter.jl` avec config par défaut.

### Architecture
- **Pas de globals mutables** (sauf le `Scheduler` qui est explicitement
  thread-safe).
- **Pas de mutation cachée** dans `Pattern` : un pattern est sémantiquement
  pur (la `Function` interne peut capturer un état, mais la query doit être
  déterministe par fenêtre — sauf `ReservoirPattern` qui est explicitement
  stateful).
- **Tests à côté du code** : chaque fichier `src/X.jl` a son `test/test_X.jl`.

### Erreurs
- Erreurs de parsing (mini-notation) → `ArgumentError` avec position.
- Erreurs OSC → `IOError` ou erreur custom `OSCError`.
- Pas d'`@assert` dans le code de production : utiliser des `throw(...)`
  explicites.

## 9. Critères de complétion globale (Phase 1)

Le projet est considéré comme **MVP complet** quand :

1. Je peux lancer `julia scripts/live.jl` et obtenir une TUI fonctionnelle.
2. Dans la TUI, écrire `d!(:d1, p"bd hh sn hh")` et entendre le pattern
   sortir de scsynth en moins de 200 ms après évaluation.
3. Modifier le pattern et entendre le changement au prochain cycle, sans
   glitch audio.
4. Combiner les combinators (`fast`, `rev`, `every`, `stack`) et obtenir le
   comportement attendu.
5. `Ctrl+H` mute tout proprement.
6. La suite de tests passe à 100% en local.

## 10. Notes pour Claude Code

### Questions à clarifier avant de coder (poser si nécessaire, sinon assumer les valeurs par défaut documentées)

1. **OSC library** : confirmer existence/version de `OpenSoundControl.jl` au
   moment du démarrage. Si non maintenue, implémenter l'encoding nous-mêmes
   (50-100 lignes, le protocole est trivial).
2. **TUI library** : `TerminalUserInterfaces.jl` est le choix par défaut.
   Alternative `Term.jl` si problème de portabilité.
3. **Format SuperDirt** : confirmer la signature exacte des messages
   `/dirt/play` en consultant le code source SuperDirt ou la doc TidalCycles
   (https://tidalcycles.org).

### Stratégie d'attaque recommandée

- **Commence par M1 + M2 en isolation** : tu peux développer et tester tous
  les combinators et la mini-notation sans aucun audio, juste avec des asserts
  sur `query(p, 0//1, 1//1)`.
- **Branche M3 progressivement** : d'abord OSC sans scheduler (envoyer un
  `/dirt/play` à la main et vérifier que ça sonne), puis le scheduler avec
  un mock OSC client (test du timing), enfin l'intégration finale.
- **TUI en dernier** : tester la jouabilité depuis le REPL Julia d'abord.

### Pièges connus à anticiper

- **Garbage collector** : peut introduire des xruns audibles. Pré-allouer
  les buffers OSC, éviter les allocations dans la hot loop du scheduler.
  Profiler avec `@allocations` au moindre doute.
- **Time-to-first-X** : la première évaluation d'un nouveau pattern paie
  un coût de compilation Julia. Documenter ce comportement, éventuellement
  pré-warm avec un pattern dummy au démarrage.
- **Double-fire d'events** : c'est le bug n°1 d'un scheduler à lookahead.
  Tracker `last_end_cycles` et ne querier que la nouvelle tranche.
- **Thread safety** : `set_pattern!` peut être appelé pendant que la boucle
  query. Protéger avec `ReentrantLock` ou utiliser un `Channel`.

### Ce qu'il NE faut PAS faire

- Pas de DSP audio en Julia en Phase 1-3. C'est scsynth qui fait le son.
- Pas de GUI. TUI uniquement.
- Pas de notebook Pluto. Scripts + TUI.
- Pas de macros métaprogrammation lourdes au-delà de `@p_str` et éventuellement
  un `@d` shorthand pour `set_pattern!`. Garder le code lisible.

## Annexe A — Exemple de session live attendue

```julia
# Démarrage (en haut du script live.jl)
using Mosaic

config = load_config("configs/default.toml")
scheduler = Scheduler(config)
start!(scheduler)

# Pendant la session — chaque ligne ci-dessous est évaluée live depuis la TUI

d!(:d1, p"bd hh sn hh")
d!(:d2, fast(2, p"cp ~ cp cp"))

# Layering avec transformation
d!(:d3, every(4, rev, p"arpy arpy:1 arpy:2 arpy:3"))

# Pattern de notes avec algèbre
d!(:d4, n_pattern(p"0 2 4 7" + p"<0 12 -12>") * sound("superpiano"))

# Mute un slot
d!(:d2, silence(Symbol))

# Phase 3 : layer reservoir
using ReservoirComputing
esn = build_esn(seed=42, n_neurons=300)
d!(:d5, ReservoirPattern(esn, note_readout, rate=8.0))

# Combine reservoir + masque rythmique manuel
d!(:d5, mask(ReservoirPattern(esn, note_readout, rate=8.0), p"1 ~ 1 1 ~ 1 ~ 1"))

# Fin de session
hush!(scheduler)
stop!(scheduler)
```
